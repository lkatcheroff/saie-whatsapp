Visión general del sistema

Tres tipos de actores interactúan con el sistema vía WhatsApp. Cada uno tiene un flujo diferenciado de mensajes entrantes y salientes.

👨‍👩‍👧

Familia

Sube documentación solicitada. Recibe notificaciones de rechazo con detalle de qué documento reenviar.

1 familia → 1 prestación
👩‍⚕️

RETAS / RET

Envía reportes, notas de voz o screenshots. Recibe notificaciones de acreditación de sueldo.

1 empleado → n prestaciones
⚙️

Sistema (SAIE)

Genera eventos automáticos: docs faltantes, acreditaciones, vencimientos.

Motor de eventos
1

Identificación única por teléfono

El número se registra una sola vez. Desde ahí el sistema reconoce automáticamente el rol en todos los mensajes siguientes. Ver panel Nuevo número.

2

Documentos trazables

Cada archivo se clasifica por tipo y se valida con Vision AI (legibilidad + verificación de contenido). Path en Storage incluye prestación, tipo y hash.

3

Audios transcritos automáticamente

Los mensajes de voz de empleados pasan por Whisper (OpenAI) antes de guardarse. El audio original queda archivado en Storage.

4

Templates aprobados para salientes

Todo mensaje iniciado por el sistema usa templates aprobados por Meta. Los mensajes dentro de la ventana de 24 hs son libres.

Primer contacto

Construcción de una conversación nueva

Cuando un número desconocido escribe por primera vez, el sistema construye su contexto de sesión antes de procesar cualquier contenido.

¿Cuándo ocurre? La primera vez que el número no está en usuarios_wa. Una vez registrado, este flujo nunca vuelve a ejecutarse para ese número.

Configuración externa

Setup de Meta WhatsApp Cloud API

Guía paso a paso con tiempos reales, trabas conocidas y cómo resolverlas. El número personal no se puede usar — se desactiva de WhatsApp al registrarlo en la API.

Tiempo activo tuyo
~1 hora de trabajo real repartida en 2 días. El resto es esperar aprobaciones de Meta.
Espera de aprobaciones
Display name: 1–3 días hábiles. Templates UTILITY: horas a 1 día. Verificación empresarial: 2–7 días (opcional al inicio).
Cuándo podés desarrollar
Desde el día 1, con el número de test de Meta. No necesitás esperar el display name para escribir código.
⚠ Número personal: al registrar un número en la Cloud API, ese número queda desvinculado de la app de WhatsApp. No se puede usar en la app y en la API al mismo tiempo (salvo modo "Coexistence", disponible en UE/UK/Sudáfrica — no Argentina todavía). Usá una SIM prepaga nueva o un número que no uses en el día a día.

Número dedicado

SIM prepaga nueva argentina (cualquier operadora, ~$500 ARS). Que pueda recibir SMS o llamada para el código de verificación. No debe estar activo en WhatsApp.

3 Display name — la principal fuente de fricción ⏱ 5 min cargar · ⏳ 1–3 días hábiles de espera ›
Reglas clave para que se apruebe al primer intento
✓ Debe coincidir exactamente con el nombre en el sitio web del Centro
✓ Sin emojis, sin signos de puntuación extra, sin mayúsculas completas
✓ Sin palabras genéricas ("Soporte", "Atención", "Info")
✓ Mínimo 3 caracteres

5 Configurar el webhook ⏱ 20 min · Sin espera ›
Endpoint Node.js
// GET /webhook — verificación inicial por Meta
app.get('/webhook', (req, res) => {
  const mode      = req.query['hub.mode'];
  const token     = req.query['hub.verify_token'];
  const challenge = req.query['hub.challenge'];
  if (mode === 'subscribe' && token === process.env.WEBHOOK_VERIFY_TOKEN) {
    res.status(200).send(challenge);
  } else { res.sendStatus(403); }
});

// POST /webhook — mensajes entrantes
app.post('/webhook', (req, res) => {
  res.sendStatus(200);        // responder 200 INMEDIATAMENTE
  processMessage(req.body);   // procesar en background
});
⚠ Trabas posibles
// TRABA 1: Webhook no verifica (403)
// La causa más común: el WEBHOOK_VERIFY_TOKEN no coincide exactamente.
// Es case-sensitive. Un espacio al final lo rompe.
// Solución: copiar el token del .env directamente, sin espacios.

// TRABA 2: Meta deja de enviar mensajes al webhook
// Ocurre si el webhook devuelve errores 5xx más de cierta cantidad de veces.
// Meta suspende temporalmente las entregas y las encola.
// Solución: SIEMPRE responder 200 inmediatamente, procesar en background.
// Nunca hacer lógica pesada antes de responder al webhook.

// TRABA 3 (2026): BSUID reemplaza phone_number
// Meta está migrando el campo `from` de phone_number a Business-Scoped User ID.
// Los campos `wa_id` y `phone_number` pueden venir null en nuevas integraciones.
// Solución: mapear BSUID → usuario en usuarios_wa además del phone.

// Para desarrollo: usar ngrok o tunnelmole para exponer localhost.
// ngrok http 3000 → obtiene URL pública HTTPS temporal.

8 Checklist final + errores comunes en producción Antes de hacer el go-live con usuarios reales ›
Checklist antes de go-live
✓ App en modo Live (no Development — en Development solo llegan msgs de testers)
✓ Número de producción en estado CONNECTED en WhatsApp Manager
✓ System user token guardado como variable de entorno en el servidor de SAIE
✓ Webhook verificado, suscrito al campo messages, respondiendo 200 en <1s
✓ Al menos 1 template UTILITY aprobado
✓ Display name aprobado (o en revisión — no bloquea el envío)
✓ Dominio con TLS activo en producción (ngrok solo para dev)
Errores frecuentes en producción (2026)
// Error 190: Token inválido o expirado → renovar token del system user
// Error 100: Template no encontrado o no aprobado → verificar nombre exacto del template
// Error 130429: Rate limit superado → throttle de envíos, max 80 msg/seg
// Webhook silencioso: Meta deja de enviar → revisar que el endpoint responde 200
// BSUID null/phone_number null: migración de Meta a BSUIDs en 2026
//   → usar el campo `user_id` del webhook además de `from`
Resumen real de tiempos: Podés tener el desarrollo funcionando con el número de test en el día 1. El número de producción propio tarda 1–3 días (display name). La verificación empresarial suma otros 2–7 días pero no bloquea — podés lanzar con el límite de 250 msgs/día mientras se procesa. El proceso completo hasta producción sin trabas es 3–5 días hábiles de espera, con ~1 hora de trabajo tuyo.

🔓 Vendor lock-in y portabilidad Qué pasa si querés cambiar de proveedor. Qué se pierde, qué se preserva. › 
Lo que SIEMPRE es tuyo (portable entre cualquier BSP o Meta directo):
  ✓ El número de teléfono
  ✓ El WABA (WhatsApp Business Account ID)
  ✓ Los templates aprobados (con calidad APPROVED)
  ✓ El display name y quality rating del número
  ✓ El blue tick / Official Business Account status
  
Lo que NO migra:
  ✗ Historial de conversaciones (queda en el BSP de origen)
  ✗ Templates con calidad LOW o REJECTED
  ✗ Automatizaciones / workflows del BSP (son propietarios)
  ✗ Contactos enriquecidos en la plataforma

Proceso de migración BSP → BSP:
  1. Deshabilitar 2FA en el número actual
  2. Embedded Signup en el nuevo BSP
  3. Verificar código SMS/voz
  4. Migración completa en minutos, sin downtime

Ventaja de Meta directo o Kapso:
  Tu WABA está bajo tu propio Business Manager.
  Migrar es mínimo — solo cambiás endpoints en el código.
  
Riesgo real con plataformas cerradas (Wati, Respond.io):
  El historial de conversaciones queda atrapado.
  Los flows/chatbots no son exportables.
  Dependencia funcional alta aunque el número sea portable.
Regla clave para elegir: BSP Tech Provider (Meta directo, Kapso, 360dialog) = WABA bajo tu Business Manager, tarifa Meta directa, portabilidad total. BSP Solution Partner tradicional (Twilio, Wati, Bird) = ellos actúan de intermediarios, pueden agregar markup, pueden dificultar la salida. Para SAIE, el camino Meta directo o Kapso es el correcto: datos en tu WABA, cero markup, máxima portabilidad.

🎯 Precisión — lo que WER no te dice Por qué los benchmarks no sirven y cómo testear en tu audio real › 
Problema con los benchmarks públicos (LibriSpeech, FLEURS):
  Audio limpio, hablantes nativos, sin ruido, sin acento local.
  Un WER de 5% en benchmark puede convertirse en 15–20% en tu audio real.
  
Factores específicos de SAIE que afectan el WER real:
  • Acento rioplatense (LL/Y, voseo, entonación)
  • Terminología clínica-educativa: "prestación", "legajo", "integración"
  • Audios de WhatsApp (ogg/m4a): compresión lossy, posible ruido ambiente
  • Grabaciones cortas (30s–3min): menos contexto para el modelo

Solución práctica:
  1. Grabar 20 audios de prueba con empleados reales de SAIE
  2. Transcribirlos en paralelo con Whisper + ElevenLabs Scribe + Gladia
  3. Comparar WER manual (contar errores en términos clave)
  4. Elegir el que mejor maneje los términos del dominio

Feature clave para SAIE: Keyterm Prompting
  ElevenLabs y AssemblyAI permiten dar un glosario de términos

  Ejemplo: ["RETAS", "RET", "prestación", "legajo", "Paso a Paso"]
  Esto puede ser decisivo en el dominio de SAIE.

🔓 Lock-in STT — es mínimo entre los players principales Por qué cambiar de proveedor STT es fácil comparado con cambiar de WA provider › 
El lock-in en STT es bajo  porque:
  • Todos tienen el mismo shape de request: audio input → text output
  • El cambio es 3–5 líneas de código (endpoint + auth + params)
  • No hay "historial atrapado" — el texto transcripto queda en tu BDD
  • Los audios originales quedan en tu Storage (Supabase)

Excepción: features propietarios
  Si dependés de AssemblyAI LeMUR (análisis semántico) → más acoplamiento
  Si dependés de Kapso Workflows (WA) → más acoplamiento
  Regla: usar features estándar de transcripción, no features AI avanzados,
  permite migrar en horas.

  Abstraer el provider en una función:
  
  async transcribeAudio(buffer, mimeType) {
    return whisperTranscribe(buffer);  // swap fácil
  }
  
  Si el día de mañana querés cambiar a Scribe o Gladia:
  modificás 1 función, no el resto del sistema.
